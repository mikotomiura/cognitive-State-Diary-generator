# 要件定義: 生成時正規化項の追加 (v6)

## 背景

v1-v5 の実験データにより、品質改善の主力は「生成時の制約注入（Pipeline → Actor Prompt）」であることが実証された。
Critic の事後スコアリング精密化（v2-v4）は品質への寄与が限定的だった。

v5 で解消されなかった残存課題:
1. 余韻末尾の構文パターンが「〜だろう」系に偏る（5-6/7 日で出現）
2. 主題語「効率」が7日間で合計22回と過剰
3. 修辞疑問文の Day 間重複リスク（v5 で 0 だが仕組みとして防止されていない）

## 実装内容

3つの新しい正規化項を Pipeline → Actor の生成時注入として追加する:

1. **余韻末尾パターンカウンター**: 余韻の構文パターン（〜だろう系/〜かもしれない系/〜ずにいる系/その他）を分類・追跡し、同一パターン3回以上の使用を禁止
2. **主題語頻度カウンター**: 主題語（効率/非効率/最適化/自動化）の累計使用回数を追跡し、閾値超過時にプロンプトで制限を注入
3. **修辞疑問文パターン追跡**: 本文中の修辞疑問文を抽出・蓄積し、同一構文の再使用を防止

## 受け入れ条件

- [ ] `_detect_ending_pattern()` が余韻の構文パターンを正しく分類する
- [ ] `_count_theme_words()` が主題語の出現回数を正しくカウントする
- [ ] `_extract_rhetorical_questions()` が修辞疑問文を正しく抽出する
- [ ] 3つの追跡変数が `PipelineRunner.run()` で Day 間で蓄積される
- [ ] `Actor.generate_diary()` に3つの新パラメータが追加される
- [ ] 追跡データがプロンプトに正しく注入される
- [ ] 全ユニットテストが Pass する
- [ ] mypy --strict でエラーがない
- [ ] ruff check でエラーがない
- [ ] Critic の変更はなし
