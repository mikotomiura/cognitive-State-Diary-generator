# Requirement — tuning-rotation-throughput

## 背景

プロンプトチューニングのサイクル（プロンプト編集 → 7 Day 生成 → Critic スコア / 出力評価 → 再編集）が
重く、回転率が上がらない。前タスク `20260507-best-of-n-parallel` で「Best-of-N 並列化」を検討した際、
真の動機が **「プロンプトチューニング回転率の向上」** であると判明し、並列化はその手段の一候補に
過ぎないことが明確になった。

並列化 / キャッシュ / 固定 seed / Day 並列 / dry-run 強化など複数の候補があり、
ROI を比較せずに 1 つの手段に飛びつくと無駄なリファクタを招く可能性が高い。
そのため、まず **計測で現状ボトルネックを定量化し、データを根拠に採用案を決定** する。

## ゴール

- [ ] 現状の 7 Day 生成サイクルにおける所要時間内訳を計測する（Phase 1 / 2 / 3 別、Day 別、retry 別）
- [ ] retry 頻度と Critic フィードバック連鎖の実効果を定量化する（retry 経由で品質改善するケースの割合）
- [ ] 同一入力での再生成頻度を計測する（プロンプト編集後の再走時に同じ Phase 結果が出る割合）
- [ ] 候補手段（Best-of-N 並列化 / LLM レスポンスキャッシュ / 固定 seed / Day 並列 / dry-run 強化）を
      ROI（実装コスト × 期待効果）で比較し、採用案を決定する
- [ ] 採用案（または採用順）を `design.md` の「採否決定」表として残し、実装は別タスクへ引き継ぐ

## 非ゴール

- 採用案の **実装** — 本タスクは計測・比較・採用決定までで、実装は別タスクで行う
- LLM クライアント側のリトライ／レート制御の刷新
- プロンプト品質の改善（チューニング作業そのものは別軸）
- 本番環境のウォールクロック短縮（動機はあくまで「チューニング回転率」）
- API コスト削減（副次効果として見るが目標ではない）

## 制約 / 前提条件

- 計測のためのコード追加は許容（pipeline.py への計測フック / 集計スクリプトなど）。
  ただし `schemas.py` の破壊的変更は禁止（CLAUDE.md 禁止事項）
- 計測スクリプトは `scripts/` または `tests/perf/` 等に配置し、本番パイプラインを汚染しない
- 既存テスト 531 件（`pytest -m "not e2e"`）は緑のまま
- 計測は手元ローカル実行のみで完結すること（CI への組み込みは別タスク）
- 前タスク `20260507-best-of-n-parallel/design.md` の Option B/C/D 分析を参考資料として再利用
