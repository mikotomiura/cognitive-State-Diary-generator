# Requirement — best-of-n-parallel

**ステータス: クローズ（実装に至らず、後続タスクへ引き継ぎ） — 2026-05-08**

`/implement refactor` 開始時の調査により、現行コードは典型的な Best-of-N（N 候補並列生成 → 最高選択）
ではなく **逐次リトライ + Critic フィードバック連鎖 + フォールバック時のみ Best-of-N 選択** であることが
判明した。単純な `asyncio.gather` 並列化は振る舞い変更を伴うため、本要件は保留としていた。

その後の方針再確認で、並列化の動機が **「プロンプトチューニング回転率の向上」** であると判明。
Best-of-N 並列化はその手段の一候補に過ぎず、他に高 ROI な手段（LLM レスポンスキャッシュ／固定 seed／
Day 並列／dry-run 強化など）が同列で比較対象になるため、**本タスクは一旦クローズ** し、
後続で広いスコープの調査タスク（例: `tuning-rotation-throughput`）を別途立ち上げる方針となった。

詳細は `design.md` の「最終結論」節を参照。

## 背景

`csdg/pipeline.py` の Best-of-N ロジックは、N 個の候補を逐次的に生成している（推測）ため、N に比例して
ウォールクロック時間が伸びる。LLM API 呼び出しは I/O バウンドであり、`asyncio.gather` 等で並列化すれば
N 候補を実質 1 候補分の待ち時間で生成できる。生成時間短縮はチューニングサイクルの回転率に直結するため、
プロンプト改善作業の生産性向上にも寄与する。

## ゴール

- [ ] Best-of-N の候補生成を asyncio で並列化し、ウォールクロック生成時間を短縮する
- [ ] 並列化前後で **生成結果の振る舞いが等価** であることを保証する（同一シードで同一の選択結果になる、
  または並列化に伴う非決定性がスコア比較で吸収されることをテストで担保）
- [ ] 既存の last-write-wins バグ（過去のコミット履歴 `bbc75b3` 周辺で修正済と推測）が
  並列環境で再発しないことを回帰テストで保証する

## 非ゴール

- 並列度のチューニング可能化（config.py への新パラメータ追加） — 今回は固定 N で並列化のみ
- LLM クライアント側のリトライ／レート制御の刷新 — 既存の挙動を維持
- Critic 評価フェーズの並列化 — Best-of-N 候補生成のみがスコープ
- プロンプト品質の改善

## 制約 / 前提条件

- `csdg/pipeline.py` の Best-of-N 部分のみを対象とする（State Update / Critic 評価フェーズには触らない）
- `schemas.py` の破壊的変更は禁止（CLAUDE.md の禁止事項）
- LLM クライアントが既に async 対応であることを前提とする — 未対応なら設計フェーズで方針判断
- 性能要件: N=3 で逐次比 おおむね 2x 以上の高速化（API レイテンシ依存だが目安）
- 後方互換性: pipeline の外部公開 API シグネチャは変更しない（main.py / tests からの呼び出しが破綻しない）
