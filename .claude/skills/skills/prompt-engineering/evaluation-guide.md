# プロンプト評価・改善ガイド

> プロンプトの品質を評価し、系統的に改善するための手法。

---

## 1. 品質評価の観点

### 1.1 出力の安定性

同じ入力で複数回実行し、出力のばらつきを確認する。

| 指標 | 理想 | 問題のサイン |
|---|---|---|
| CriticScore の標準偏差 | 0.5 以下 | 1.0 超 → プロンプトが曖昧 |
| 1回で Pass する割合 | 80% 以上 | 50% 未満 → 指示が不十分 |
| リトライ後に Pass する割合 | 95% 以上 | 80% 未満 → フォールバックに依存しすぎ |

### 1.2 指示の具体性

プロンプト内の指示が以下の基準を満たしているか確認する:

- **具体的か?** 「良い文章を書いて」→ ❌ / 「壮大な比喩を含め、余韻記号 ...... で締める」→ ✅
- **測定可能か?** 「感情を表現して」→ ❌ / 「stress が 0.5 以上の場合、短文連打を3文以上含める」→ ✅
- **例示があるか?** 抽象的な指示のみ → ❌ / 具体例（良い例・悪い例）付き → ✅

### 1.3 ペルソナの一貫性

Day 1〜7 を通じて以下が維持されているか:

- 一人称が「わたし」で統一されている
- 絵文字が一切使用されていない
- 結論が断定されていない
- 比喩のスタイルがキャラクターと一致している

## 2. 問題の診断フロー

```
CriticScore が低い
  ├─ temporal_consistency が低い
  │   ├─ memory_buffer が参照されていない → Prompt_StateUpdate.md を修正
  │   └─ 過去との矛盾がある → Prompt_Generator.md に「過去の記憶を参照」の指示を強化
  │
  ├─ emotional_plausibility が低い
  │   ├─ 感情変化が小さすぎる → Prompt_StateUpdate.md の変動ルールを具体化
  │   ├─ 感情変化が大きすぎる → EMOTION_SENSITIVITY の調整を検討
  │   └─ deviation が大きい → expected_delta の計算式をプロンプトに明示
  │
  └─ persona_deviation が低い
      ├─ 禁則事項の違反 → System_Persona.md に「こう書いてはいけない」例を追加
      ├─ キャラクターらしさが薄い → Prompt_Generator.md に文体の具体例を追加
      └─ 汎用的な表現が多い → Few-shot で「とこみらしい」具体例を追加
```

## 3. 改善テクニック

### 3.1 Few-shot 例の追加

LLM は抽象的な指示より具体例から学ぶ。

```markdown
## 文体の例

### 良い例（とこみらしい）
「今日のスクラムミーティングは、まるでソクラテスが市場で繰り広げた対話の
21世紀版だった。ただし、ソクラテスは『なぜこの機能が必要なのか』を問うたら
毒杯を渡されたわけだけど、わたしの場合はJIRAチケットを渡されただけマシ
......なのかもしれない。」

### 悪い例（避けるべき）
「今日のスクラムミーティングは有意義でした。チームで議論を重ね、
良い結論に至りました。」
→ NG: 断定している、比喩がない、余韻がない、汎用的すぎる
```

### 3.2 否定形から肯定形への変換

```markdown
# ❌ 否定形（LLMが「何をするか」が分かりにくい）
- 結論を断定しないでください
- 絵文字を使わないでください

# ✅ 肯定形（「何をするか」が明確）
- 文章の最後は「......なのかもしれない」「分からないけど」等の
  問いかけや余韻で締めてください
- 感情は言葉と比喩のみで表現してください（記号や絵文字は使用しません）
```

### 3.3 段階的な指示の精緻化

一度に大きく変えず、1つずつ変更してスコアへの影響を計測する:

```
変更1: persona_deviation 改善のため、禁則事項の例を追加
  → Day 1-7 実行、persona_deviation の変化を計測
  → 改善 → 次の変更へ / 悪化 → revert

変更2: emotional_plausibility 改善のため、emotional_impact の大きさに応じた文体指示を追加
  → Day 4 (emotional_impact=-0.9) で重点的に計測
  → 改善 → 次の変更へ / 悪化 → revert
```

### 3.4 Critic の調整

Critic のプロンプトが厳しすぎる/甘すぎる場合:

**厳しすぎる場合（リトライが多発）:**
- スコア3の基準を「最低限の品質を満たしている」に緩和する
- deviation の許容範囲を広げる

**甘すぎる場合（品質が低いのに Pass する）:**
- 各スコアの具体的な Pass/Reject 例を追加する
- 「以下の条件に1つでも該当すれば2以下にする」等の明確なルールを追加する

## 4. 変更の記録

プロンプト変更は `.steering/` の `decisions.md` に以下を必ず記録する:

```markdown
## [日時] Prompt_Generator.md — 文体指示の強化

### 変更内容
「感情爆発時」の文体指示に具体例を2つ追加した。

### 変更前スコア（Day 4）
- temporal: 4, emotional: 2, persona: 3 → Reject

### 変更後スコア（Day 4）
- temporal: 4, emotional: 4, persona: 4 → Pass

### 判定
✅ 採用 — emotional_plausibility が 2→4 に改善。他Dayへの悪影響なし。
```
