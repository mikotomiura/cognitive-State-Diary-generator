# /setup-docs — 永続ドキュメント構築コマンド

> Phase 1 of 7. Let's think step by step.
> このコマンドは後続のすべての構築物の基盤となるドキュメントを作成する。
> 妥協しない。

## 環境チェックブロック（必須）

### Check 1: 進捗ファイルの確認

```bash
cat .steering/_setup-progress.md
```

ファイルが存在しない場合は中断し、「`/bootstrap` を先に実行してください」と通知。

進捗ファイルから「Phase 0: Bootstrap」が完了マークされていることを確認。されていない場合は中断。

### Check 2: コンテキスト予算

ユーザーに `/context` の実行を依頼。**使用率が 30% を超えていたら中断**し、`/clear` を促す。

### Check 3: モデル

ユーザーに `/model` で現在のモデルを確認するよう依頼。**Opus でなければ強い警告**:

> 「設計判断には Opus が必要です。`/model opus` に切り替えてください。Sonnet で続行する場合、ドキュメントの品質が大幅に下がる可能性があります。」

ユーザーが明示的に Sonnet を選択した場合のみ進む。

### Check 4: Plan Mode

Plan Mode に入っているかを確認。入っていない場合は `Shift+Tab` 2 回での切り替えを推奨。

### Check 5: docs/ ディレクトリの状態

```bash
ls docs/ 2>/dev/null
```

既存ファイルがある場合、ユーザーに「上書きしますか?マージしますか?スキップしますか?」と確認。

## 実行フロー

### Step 1: プロジェクト調査（並列サブエージェント）

> ⚠️ サブエージェント (`file-finder`, `dependency-checker`, `impact-analyzer`) は Phase 4 で作成されるため、**この時点ではメインエージェントが直接調査を行う**。

メインエージェントが以下を順番に実行:

#### 1.1 ファイル構造の把握

```bash
find . -type f -not -path '*/node_modules/*' -not -path '*/.git/*' -not -path '*/dist/*' -not -path '*/build/*' | head -50
```

主要なソースディレクトリの存在を確認。

#### 1.2 設定ファイルの読み込み

存在するものすべてを Read で読む:
- `package.json`
- `pyproject.toml`
- `Cargo.toml`
- `requirements.txt`
- `tsconfig.json`
- `.eslintrc.*`
- `pytest.ini`

#### 1.3 README の確認

```bash
cat README.md 2>/dev/null
```

#### 1.4 既存ドキュメントの確認

```bash
find docs/ -type f -name "*.md" 2>/dev/null
```

存在する場合はすべて Read で読む。

#### 1.5 主要なソースファイルのサンプリング

```bash
find src/ -type f \( -name "*.py" -o -name "*.ts" -o -name "*.tsx" -o -name "*.rs" \) | head -10
```

最大 10 ファイルだけ Read で確認。深く読まず、構造だけ把握。

### Step 2: 調査結果の統合とユーザーへの追加ヒアリング

Step 1 の調査結果をもとに、`.steering/_setup-progress.md` の「プロジェクト概要」を補完するための追加質問をユーザーに投げる:

1. 調査の結果、このプロジェクトは [推測内容] と理解しました。正しいですか?
2. 主要なユースケースを 3 つ挙げてください
3. 採用しているアーキテクチャパターン（MVC, Clean Architecture, etc.）はありますか?
4. 特殊な制約や規約（業界固有のもの、社内ルール）はありますか?

回答を待つ。

### Step 3: 5 つのドキュメントを 1 つずつ作成

> **重要**: 必ず 1 つずつ作成し、各ファイルの作成後に **(a) ユーザーの承認** と **(b) Grill me ステップ** を実施してから次に進む。

各ドキュメントの作成プロセスは以下の 4 段階で進める:

1. **ドラフト作成**
2. **ユーザーへの提示と承認**
3. **Grill me ステップ（自己批判的レビュー）**
4. **最終化**

### Step 4: 全体整合性レビュー

5 つすべてのドキュメントが完成したら、全体を通しで自己レビューする:

> 5 つのドキュメントを横断的に確認します:
> - functional-design と architecture の間に矛盾はないか?
> - architecture と repository-structure の構造は一致しているか?
> - development-guidelines のテスト方針が repository-structure のテストディレクトリ構造と整合しているか?
> - glossary に登場する用語が他の 4 ファイルで実際に使われているか?
> - 5 つのドキュメント全体で同じ概念に異なる名前を使っていないか?

問題があれば修正する。

### Step 5: 進捗ファイルの更新

```markdown
- [x] **Phase 1: /setup-docs** — 永続ドキュメント
  - 完了日時: [YYYY-MM-DD HH:MM]
```
