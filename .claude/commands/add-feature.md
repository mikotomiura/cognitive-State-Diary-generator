# /add-feature — 新機能追加ワークフロー

> **目的:** CSDG プロジェクトに新しい機能を追加する際の、調査から実装・テスト・ドキュメント更新までの完全なワークフローを定義する。
> **粒度:** 1回の作業セッションで完了する1つの機能追加。

---

## ステップ 1: プロジェクト理解

1. `CLAUDE.md` を読み、プロジェクトの全体像と開発原則を確認する
2. `docs/glossary.md` を読み、関連する用語を確認する
3. `docs/functional-design.md` を読み、追加する機能が既存の機能要件と整合するか確認する
4. `docs/architecture.md` を読み、変更対象のモジュールと依存関係を確認する

**確認事項:**
- この機能は `functional-design.md` のスコープ内か？
- 既存のモジュール構成で実現可能か、新規モジュールが必要か？
- `glossary.md` に新しい用語を追加する必要があるか？

---

## ステップ 2: 作業記録の開始

1. `.steering/[YYYYMMDD]-[機能名]/` ディレクトリを作成する
2. `requirement.md` を作成し、以下を記述する:
   - **背景:** なぜこの機能が必要か
   - **実装内容:** 何を実装するか（具体的な振る舞い）
   - **受け入れ条件:** 機能が完成したと判断するための条件リスト
   - **影響範囲:** 変更が影響するモジュール・ファイル
3. `design.md` を作成し、以下を記述する:
   - **実装アプローチ:** どのように実装するか
   - **変更対象ファイル:** ファイルパスと変更内容のテーブル
   - **代替案と選定理由:** 検討した代替案とその却下理由
4. `tasklist.md` を作成し、実装・テスト・ドキュメントの具体的なタスクをチェックリストで記述する

---

## ステップ 3: 既存パターンの調査

1. `impact-analyzer` サブエージェントを起動し、変更の影響範囲を分析する
2. 変更対象モジュールの既存コードを読み、以下を確認する:
   - 既存のコーディングパターン（エラーハンドリング、ロギング、型アノテーション）
   - 既存テストのパターン（フィクスチャ、モック、パラメタライズ）
   - 既存のPydanticモデルのパターン（Field定義、バリデータ）
3. `file-finder` サブエージェントを起動し、類似する既存実装を検索する
4. 既存パターンに倣った実装方針を決定する

---

## ステップ 4: ドキュメントの先行更新

**設計書ファーストの原則に従い、コードより先にドキュメントを更新する。**

1. `docs/glossary.md` — 新しい用語があれば追加する
2. `docs/functional-design.md` — 機能要件（F-XX-XX）を追加する
3. `docs/architecture.md` — モジュール設計・データフローに変更があれば更新する
4. `docs/repository-structure.md` — 新規ファイルがあればツリーとモジュール一覧に追加する

---

## ステップ 5: 実装

1. **ブランチを作成する:**
   ```bash
   git checkout -b feat/[機能名]
   ```

2. **Pydantic モデルの変更（必要な場合）:**
   - `csdg/schemas.py` にフィールド/モデルを追加する
   - `Field(description=...)` を付ける
   - `field_validator` で必要なバリデーションを実装する
   - 連続変数は `-1.0` 〜 `1.0` にクランプする

3. **メインロジックの実装:**
   - `docs/development-guidelines.md` のコーディング規約に従う
   - すべての関数に型アノテーションを付ける
   - Google style docstring を記述する
   - マジックナンバーは `config.py` で管理する
   - プロンプトはコードに埋め込まず `prompts/` で管理する
   - エラーハンドリングとフォールバックを実装する
   - `logging` モジュールでログを記録する

4. **プロンプトの更新（必要な場合）:**
   - `prompts/` のMarkdownファイルを更新する
   - プレースホルダ名がコードと一致していることを確認する
   - `glossary.md` の用語と一致していることを確認する

---

## ステップ 6: テストの実装

1. **テストファイルの作成/更新:**
   - 被テストモジュールに対応する `tests/test_*.py` を作成する
   - 共通フィクスチャは `tests/conftest.py` に追加する

2. **テストケースの実装:**
   - 正常系テスト: 期待通りの入力で期待通りの出力が得られるか
   - 境界値テスト: `-1.0`, `0.0`, `1.0` および範囲外の値
   - 異常系テスト: `ValidationError`, API エラー, タイムアウト
   - パラメタライズ: `@pytest.mark.parametrize` で類似ケースを統合

3. **LLM API のモック:**
   - `LLMClient` の抽象インターフェースに対してモックする
   - 正常系と異常系の両方のモックを準備する

---

## ステップ 7: テストと検証

1. **テストの実行:**
   ```bash
   pytest tests/ -v --cov=csdg --cov-report=term-missing -m "not e2e"
   ```

2. **型チェック:**
   ```bash
   mypy csdg/ --strict
   ```

3. **リンター:**
   ```bash
   ruff check csdg/
   ruff format csdg/ --check
   ```

4. **カバレッジ確認:**
   `development-guidelines.md` のカバレッジ目標を満たしているか確認する

5. **問題があれば修正し、再度 Step 7 を実行する**

または、`build-executor` サブエージェントを起動して一括検証する。

---

## ステップ 8: ドキュメント最終更新

1. `.steering/` の `tasklist.md` を完了状態に更新する
2. 重要な決定があれば `decisions.md` に記録する
3. `CLAUDE.md` に影響がある場合は更新する（サブエージェント・コマンド・スキル一覧等）

---

## ステップ 9: コミットとレビュー

1. **コミット:**
   ```bash
   git add .
   git commit -m "feat([scope]): [変更内容の要約]"
   ```

2. **セルフレビュー:**
   `code-reviewer` サブエージェントを起動し、変更内容をレビューする

3. **プッシュ:**
   ```bash
   git push origin feat/[機能名]
   ```

4. **Pull Request の作成:**
   `development-guidelines.md` のPRテンプレートに従って作成する

---

## チェックリスト（完了前の最終確認）

- [ ] `glossary.md` の用語を正しく使用している
- [ ] すべての関数に型アノテーションがある
- [ ] すべての公開関数に docstring がある
- [ ] マジックナンバーが存在しない
- [ ] プロンプトがコード内に埋め込まれていない
- [ ] テストが存在し、全件 Pass している
- [ ] mypy --strict でエラーがない
- [ ] ruff check でエラーがない
- [ ] カバレッジ目標を満たしている
- [ ] 影響を受けるドキュメントを更新した
- [ ] `.steering/` に作業記録を残した
- [ ] `repository-structure.md` を更新した（新規ファイル追加時）
