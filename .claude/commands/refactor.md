# /refactor — リファクタリングワークフロー

> **目的:** 外部的な振る舞いを変えずに、コードの内部構造を改善する。
> **粒度:** 1つのリファクタリング目的（関数抽出、モジュール分割、パターン統一等）に集中する。
> **鉄則:** リファクタリング中に機能追加やバグ修正を混ぜない。

---

## ステップ 1: リファクタリングの動機確認

1. **対象コードの確認:**
   - リファクタリング対象のファイル・関数・クラスを Read で読み込む
   - 現在の問題点を具体的に特定する（長すぎる関数、重複コード、不明瞭な命名、深いネスト等）

2. **リファクタリングの種別を決定する:**

   | 種別 | 例 | 判断基準 |
   |---|---|---|
   | 関数抽出 | 50行超の関数を分割 | 関数が複数の責務を持っている |
   | モジュール分割 | 肥大化したファイルを分割 | ファイルが300行を超えている |
   | 命名改善 | 変数名・関数名の明確化 | 意図が伝わらない名前 |
   | パターン統一 | エラーハンドリングの統一 | 同じ処理が異なるパターンで書かれている |
   | 重複除去 | 共通処理のユーティリティ化 | 同じコードが3箇所以上にある |
   | 型の強化 | `dict` → 専用Pydanticモデル | 型が曖昧でバグを誘発する |

---

## ステップ 2: 作業記録と影響分析

1. `.steering/[YYYYMMDD]-refactor-[対象]/` ディレクトリを作成する
2. `requirement.md` に動機と改善目標を記述する
3. `impact-analyzer` サブエージェントを起動し、変更の影響範囲を特定する
4. `design.md` に変更対象ファイル・変更内容・影響範囲を記述する
5. `tasklist.md` にリファクタリングのステップを分解して記述する

---

## ステップ 3: テストの確認（安全ネットの構築）

**リファクタリングを開始する前に、既存テストが十分であることを確認する。**

1. `test-runner` サブエージェントを起動し、現在のテスト状態を確認する
2. 全テストが Pass していることを確認する（Red の状態でリファクタリングを始めない）
3. リファクタリング対象のコードに対するテストカバレッジを確認する
4. カバレッジが不十分な場合、**先にテストを追加してからリファクタリングに入る**

```bash
pytest tests/ -v --cov=csdg --cov-report=term-missing -m "not e2e"
```

---

## ステップ 4: リファクタリングの実行

1. **ブランチを作成する:**
   ```bash
   git checkout -b refactor/[対象]
   ```

2. **小さなステップで進める:**
   - 1つの変更を行う
   - テストを実行して Pass を確認する
   - 次の変更に進む
   - **テストが Fail したら直ちに直前の変更を見直す**

3. **各ステップで守ること:**
   - 外部インターフェース（公開関数のシグネチャ、モジュールのエクスポート）を変えない
   - 変えざるを得ない場合は、呼び出し元もすべて更新する
   - `architecture.md` のモジュール依存方向を維持する
   - 新たな循環参照を作らない

4. **ステップごとにコミットする:**
   ```bash
   git commit -m "refactor([scope]): [このステップの変更内容]"
   ```

---

## ステップ 5: 全体検証

1. **全テストの実行:**
   ```bash
   pytest tests/ -v --cov=csdg -m "not e2e"
   ```

2. **カバレッジがリファクタリング前と同等以上であることを確認する**

3. **型チェック・リンター:**
   ```bash
   mypy csdg/ --strict
   ruff check csdg/
   ```

4. `code-reviewer` サブエージェントで変更内容をレビューする

---

## ステップ 6: ドキュメント更新とコミット

1. モジュール構成が変わった場合:
   - `docs/architecture.md` のモジュール設計セクションを更新する
   - `docs/repository-structure.md` のディレクトリツリーとモジュール一覧を更新する
2. `.steering/` の `tasklist.md` を完了状態に更新する
3. Push + Pull Request を作成する

---

## チェックリスト（完了前の最終確認）

- [ ] 外部的な振る舞いが変わっていない（全テスト Pass）
- [ ] カバレッジがリファクタリング前と同等以上
- [ ] 新たな循環参照が発生していない
- [ ] mypy --strict でエラーがない
- [ ] 機能追加やバグ修正を混ぜていない
- [ ] `.steering/` に作業記録を残した
- [ ] 影響を受けるドキュメントを更新した
