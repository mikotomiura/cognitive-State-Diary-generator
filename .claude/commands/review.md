# /review — コードレビューワークフロー

> **目的:** 実装済みのコードに対して品質チェックを実行し、問題を検出・報告する。
> **粒度:** 1回のレビューセッション。対象はPull Request単位、またはファイル指定。

---

## ステップ 1: レビュー対象の特定

1. **対象の確認:**
   - Pull Request の差分ファイルを確認する
   - または、親エージェントから指定されたファイルパスを確認する

2. **変更の概要を把握する:**
   - コミットメッセージを読み、変更の意図を理解する
   - `.steering/` の対応する作業記録（`requirement.md`, `design.md`）があれば読む

---

## ステップ 2: 自動チェックの実行

1. **`build-executor` サブエージェントを起動する:**
   - ruff check, ruff format, mypy --strict, pytest の一括実行
   - 失敗がある場合は、レビューに進む前に修正を依頼する

2. **`security-checker` サブエージェントを起動する:**
   - APIキーの露出、ログへの機密情報混入をチェック

---

## ステップ 3: コードレビュー

1. **`code-reviewer` サブエージェントを起動する:**
   - 型安全性、エラーハンドリング、命名規則、設計整合性、ドキュメントを検査
   - 重大度別（Critical / Warning / Info / Good）のレポートを取得する

2. **CSDG 固有の観点を手動で確認する:**

   **パイプライン関連の変更:**
   - Phase 1/2/3 の分離が維持されているか
   - Actor と Critic の責務が混在していないか
   - Self-Healing フォールバックが実装されているか
   - Temperature Decay の設定が `config.py` で管理されているか

   **Pydantic モデルの変更:**
   - `schemas.py` の変更が `prompts/` と整合しているか
   - `field_validator` のクランプが連続変数に実装されているか
   - `memory_buffer` のサイズ制限が維持されているか

   **プロンプトの変更:**
   - `glossary.md` の用語と一致しているか
   - ペルソナの禁則事項（絵文字禁止、断定禁止等）が維持されているか
   - プレースホルダ名がコードと一致しているか

---

## ステップ 4: テスト品質の確認

1. **`test-analyzer` サブエージェントを起動する:**
   - 新規コードに対応するテストが存在するか
   - カバレッジ目標を満たしているか
   - テストの品質（境界値、異常系、パラメタライズ）は十分か

---

## ステップ 5: レビューレポートの作成

以下のフォーマットで統合レポートを作成する:

```markdown
# 統合レビューレポート

## 対象
- ブランチ / PR: (対象)
- レビュー日時: YYYY-MM-DD HH:MM

## 自動チェック結果
| チェック | 結果 |
|---|---|
| ruff check | ✅ / ❌ |
| mypy --strict | ✅ / ❌ |
| pytest | ✅ / ❌ |
| セキュリティ | ✅ / ❌ |

## コードレビュー結果
- 🔴 Critical: X 件
- 🟡 Warning: X 件
- 🔵 Info: X 件

## テスト品質
- カバレッジ: XX%（目標: XX%）
- 不足テスト: (具体的な指摘)

## 総合判定
- ✅ **承認** — マージ可能
- 🟡 **条件付き承認** — Warning の対応後にマージ可能
- 🔴 **要修正** — Critical の修正が必要

## 詳細な指摘事項
(code-reviewer のレポートから転記)
```

---

## チェックリスト（レビュー完了前の確認）

- [ ] `code-reviewer` サブエージェントを実行した
- [ ] CSDG 固有の観点を確認した
- [ ] テスト品質を確認した
- [ ] 総合判定を記述した
- [ ] 指摘事項に具体的な修正案を含めた
