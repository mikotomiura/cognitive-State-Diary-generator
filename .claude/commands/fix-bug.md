# /fix-bug — バグ修正ワークフロー

> **目的:** バグの再現・原因特定・修正・検証の一連のプロセスを TDD（Red-Green-Refactor）で実行する。
> **粒度:** 1つのバグに対する修正。複数のバグを同時に修正しない。

---

## ステップ 1: バグの理解と再現

1. **バグの情報を収集する:**
   - 報告されたバグの症状（何が起きているか）
   - 期待される動作（何が起きるべきか）
   - 再現手順（どうすれば再現できるか）

2. **関連コードの確認:**
   - `file-finder` サブエージェントで関連ファイルを検索する
   - 該当モジュールのコードを読み、バグが発生する箇所を特定する

3. **バグの再現:**
   - 既存のテストで再現可能か確認する
   - 再現できない場合、手動で再現手順を実行する
   - `generation_log.json` が存在する場合、`log-analyzer` サブエージェントで関連する異常を特定する

---

## ステップ 2: 作業記録の開始

1. `.steering/[YYYYMMDD]-fix-[バグ名]/` ディレクトリを作成する
2. `requirement.md` を作成し、以下を記述する:
   - **バグの症状:** 何が起きているか
   - **期待される動作:** 何が起きるべきか
   - **再現手順:** どうすれば再現できるか
   - **影響範囲:** このバグが影響する機能・モジュール
   - **受け入れ条件:** バグが修正されたと判断する条件
3. `design.md` を作成し、原因分析と修正アプローチを記述する

---

## ステップ 3: テストの追加（Red）

**バグを修正する前に、バグを再現するテストを書く。**

1. 該当するテストファイル（`tests/test_*.py`）を特定する
2. バグの症状を再現するテストケースを追加する
3. テストを実行し、**テストが失敗すること（Red）を確認する**

```bash
pytest tests/test_[対象].py::[テストクラス]::[テストメソッド] -v
```

**重要:** このステップでテストが Pass してしまう場合、バグの理解が不十分か、テストの書き方が不適切。Step 1 に戻る。

---

## ステップ 4: 原因の特定と修正（Green）

1. **根本原因の特定:**
   - `impact-analyzer` サブエージェントで影響範囲を確認する
   - バグの直接原因（どのコードが問題か）を特定する
   - バグの根本原因（なぜそのコードが問題になったか）を特定する

2. **最小限の修正を実装する:**
   - バグの根本原因のみを修正する（ついでの修正をしない）
   - `development-guidelines.md` のコーディング規約に従う
   - 型アノテーション、docstring、エラーハンドリングを確認する

3. **テストが Pass することを確認する（Green）:**
   ```bash
   pytest tests/test_[対象].py::[テストクラス]::[テストメソッド] -v
   ```

---

## ステップ 5: リファクタリング（Refactor）

1. 修正したコードの品質を確認する:
   - 同じバグが他の箇所で発生する可能性はないか
   - 修正がコードの明瞭性を損なっていないか
   - 防御的なバリデーションを追加すべきか

2. 必要に応じてリファクタリングする（テストが Pass し続けることを確認しながら）

---

## ステップ 6: 全体検証

1. **全テストの実行:**
   ```bash
   pytest tests/ -v --cov=csdg -m "not e2e"
   ```

2. **型チェック:**
   ```bash
   mypy csdg/ --strict
   ```

3. **リンター:**
   ```bash
   ruff check csdg/
   ```

4. **リグレッションがないことを確認する**

または、`build-executor` サブエージェントを起動して一括検証する。

---

## ステップ 7: コミットとレビュー

1. `.steering/` の `tasklist.md` を完了状態に更新する
2. コミット:
   ```bash
   git add .
   git commit -m "fix([scope]): [バグ修正の要約]"
   ```
3. `code-reviewer` サブエージェントで変更内容をセルフレビューする
4. Push + Pull Request を作成する

---

## チェックリスト（完了前の最終確認）

- [ ] バグを再現するテストを追加した
- [ ] テストが Red → Green に変化した
- [ ] 修正は最小限で、ついでの変更を含んでいない
- [ ] 全テストが Pass している（リグレッションなし）
- [ ] mypy --strict でエラーがない
- [ ] `.steering/` に作業記録を残した
- [ ] 根本原因と修正内容が `design.md` に記録されている
