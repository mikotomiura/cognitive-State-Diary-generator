# /update-docs — ドキュメント更新ワークフロー

> **目的:** ドキュメントの追加・修正を行い、コードとドキュメントの整合性を維持する。
> **粒度:** 1つのドキュメント更新理由（コード変更の反映、誤りの修正、新セクションの追加）に集中する。

---

## ステップ 1: 更新の動機と対象の特定

1. **更新の動機を確認する:**
   - コード変更に伴うドキュメントの追随（最も頻度が高い）
   - ドキュメント自体の誤りや不足の修正
   - 新しいセクション・ドキュメントの追加

2. **更新対象を特定する:**
   コード変更に伴う場合、以下の依存関係に基づいて影響を受けるドキュメントを特定する:

   ```
   glossary.md          ← 用語の追加・変更がある場合（最優先）
     ↑
   functional-design.md ← 機能要件の追加・変更がある場合
     ↑
   architecture.md      ← モジュール構成・データフロー・技術設計の変更がある場合
     ↑
   repository-structure.md ← ファイル追加・移動・削除がある場合
     ↑
   development-guidelines.md ← 規約・プロセスの変更がある場合
   ```

3. **CLAUDE.md への影響を確認する:**
   - サブエージェント・スラッシュコマンド・スキルの追加・変更
   - 技術スタックの変更
   - よく使うコマンドの変更

---

## ステップ 2: 更新の実行

**依存関係の下層（`glossary.md`）から上層に向かって更新する。**

### 2-1. `glossary.md` の更新

- 新しい用語があれば、適切なカテゴリに追加する
- 既存の用語の定義に変更があれば修正する
- 用語の変更は他のドキュメント全体に波及するため、特に慎重に行う

### 2-2. `functional-design.md` の更新

- 機能要件（F-XX-XX）の追加・変更
- ユースケースの追加・変更
- 入出力仕様の変更
- 品質基準の変更

### 2-3. `architecture.md` の更新

- モジュール設計の変更（依存関係図、責務テーブル）
- データスキーマの変更（Pydanticモデルのフィールド追加等）
- パイプラインフローの変更
- 技術選定の変更

### 2-4. `repository-structure.md` の更新

- ディレクトリツリーの更新
- モジュール一覧テーブルの更新
- 命名規則の追加
- 新規ファイル追加時のチェックリスト更新

### 2-5. `development-guidelines.md` の更新

- コーディング規約の追加・変更
- テスト方針の変更
- Git運用の変更

### 2-6. `CLAUDE.md` の更新

- ドキュメント体系テーブルの変更
- サブエージェント・コマンド・スキル一覧の変更
- コマンド一覧の変更

---

## ステップ 3: 整合性の検証

1. **用語の一致確認:**
   - `file-finder` サブエージェントで、`glossary.md` に追加/変更した用語が他のドキュメントで使用されている箇所を検索する
   - 表記ゆれがないことを確認する

2. **相互参照の確認:**
   - ドキュメント間のリンク（`[技術設計書](./architecture.md)` 等）が有効か確認する
   - 参照先のセクションが存在するか確認する

3. **コードとの整合性確認:**
   - `architecture.md` のモジュール一覧がコードの実態と一致しているか
   - `repository-structure.md` のディレクトリツリーが実際のファイル構成と一致しているか
   - `functional-design.md` の環境変数テーブルが `config.py` と一致しているか

---

## ステップ 4: コミット

```bash
git add docs/
git commit -m "docs([対象ドキュメント]): [変更内容の要約]"
```
