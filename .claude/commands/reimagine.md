## 実行フロー

### Step 1: 入力の確認

ユーザーに以下を 1 問ずつ聞く:

1. **対象**: 何を再構想するか? (例: 「Critic の評価ロジック設計」「pipeline.py の Best-of-N 実装方針」)
2. **既に検討した案があるか**: あれば 1〜3 行で要約してもらう (これが「Plan A の初回案」になる)
3. **制約の優先順位**: 速度 / 保守性 / 互換性 / テスタビリティ / 計算量 のうち主軸 1〜2 個

### Step 2: Plan A を最小限で固める

ユーザーが提示した初回案 (またはこの場で短く生成した初稿) を `.steering/[現在のタスク]/decisions.md` に **A 案** として記録する。

```markdown
## A 案 (初回案)
- アプローチ: [要約]
- メリット: [...]
- デメリット: [...]
- 主軸とした制約: [...]
```

### Step 3: 制約を意図的にずらして B 案を生成

A 案の主軸制約と **異なる軸** を主軸に据えて、B 案を独立に組み立てる。
A 案の中身に引きずられないよう、A 案を読み返さずに白紙から思考を始める。

具体的な制約変換例:
- A=「速度優先」→ B=「保守性優先」(複雑な最適化を捨てて読みやすさを取る)
- A=「最小変更」→ B=「全面リファクタ」(技術債を返済する選択肢を真面目に評価)
- A=「既存抽象を再利用」→ B=「新しい抽象を導入」
- A=「同期処理」→ B=「非同期/並列化」
- A=「インライン実装」→ B=「外部委譲 (Codex / 別モジュール)」

設計が大きい時は `Agent` (subagent_type=Plan) を起動して B 案を独立に立てさせるのも有効。

### Step 4: 並べて比較

`.steering/[現在のタスク]/decisions.md` に B 案を追記し、対比表を生成:

```markdown
## B 案 (制約を変えて独立生成)
- アプローチ: [要約]
- メリット: [...]
- デメリット: [...]
- 主軸とした制約: [...]

## 比較

| 観点 | A 案 | B 案 |
|---|---|---|
| 実装コスト | ... | ... |
| 保守性 | ... | ... |
| テスト容易性 | ... | ... |
| 既存コードへの影響 | ... | ... |
| 後戻りコスト | ... | ... |
```

### Step 5: 判定とハイブリッド検討

ユーザーに尋ねる:
> 「A 案 / B 案 / 両者の良い部分を組み合わせたハイブリッド C 案 のどれを採用しますか?」

ハイブリッドを選んだ場合、C 案を decisions.md に追記してから次へ。

### Step 6: 採用案の確定

採用案を `design.md` の正式な設計セクションへ昇格させる。
不採用案は decisions.md に残す (将来の振り返りで価値がある)。

## 制約

- A 案を生成 / 提示しないまま B 案を作らない (比較対象が無くなる)
- B 案を A 案の派生として生成しない (同じ思考の延長になり diversification 効果が消える)
- 制約軸を明示せずに「もう一度考えてみました」だけで済ませない (恣意的になる)
- decisions.md への記録を省略しない (採用根拠が後から検証不能になる)
- Plan Mode 外で安易に使わない (実装着手後の reimagine は手戻りが大きい)

## アンチパターン

- A 案と B 案が実質同じ (制約軸を変えていない兆候)
- 「両方良い」で結論を先送りする (主軸制約に基づき必ず一案を選ぶ、ハイブリッドは別案 C として扱う)
- 比較表が空欄だらけ (各観点を埋められないなら案がそもそも未成熟)
