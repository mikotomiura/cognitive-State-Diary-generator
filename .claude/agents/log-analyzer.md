# ログ分析 サブエージェント

## 目的

CSDG パイプラインの実行ログ（`output/generation_log.json`）を解析し、パイプラインの動作品質、パフォーマンス、問題箇所を特定する分析レポートを生成する。

---

## 分析観点

### 1. パイプライン健全性

- **全Day生成成功率:** 7Day中何Dayが成功したか
- **リトライ発生率:** 全Dayに対するリトライが発生したDayの割合
- **フォールバック発生率:** Best-of-N またはPhase 1フォールバックが使用されたDayの割合
- **Dayスキップ:** スキップされたDayがあるか

### 2. CriticScore 分析

- **平均スコア:** 3つの評価軸（temporal_consistency, emotional_plausibility, persona_deviation）の平均
- **最低スコア:** 各評価軸での最低スコアとそのDay
- **スコア推移:** Day 1〜7 にかけてのスコアの傾向（改善/悪化/安定）
- **Reject原因:** reject_reason の分類と頻度
- **Day 4（転機）のスコア:** emotional_impact=-0.9 に対してシステムがどう応答したか

### 3. 感情パラメータの妥当性

- **expected_delta との乖離:** 各Dayで計算された deviation が許容範囲内か
- **急激な変動:** 隣接Day間での感情パラメータの変化量が不自然に大きくないか
- **範囲逸脱:** クランプが発動した形跡がないか（元の値が -1.0〜1.0 を超えていた可能性）
- **物語アークとの整合性:** stress/motivation/fatigue の推移がシナリオ設計の想定と一致しているか

### 4. パフォーマンス分析

- **Phase別処理時間:** Phase 1/2/3 それぞれの平均・最大処理時間
- **ボトルネック:** 処理時間が突出して長い Phase/Day の特定
- **リトライによる追加コスト:** リトライが発生したDayの追加処理時間
- **API呼び出し数:** 合計および Day 別の API 呼び出し回数
- **推定コスト:** トークン数ベースでの推定 API コスト（可能であれば）

### 5. パターン検出

- **繰り返される Reject パターン:** 同じ reject_reason が複数Dayで発生していないか
- **Temperature Decay の効果:** リトライでTemperatureを下げた際にスコアが改善しているか
- **Phase 1 バリデーションエラー:** フォールバックが使用された場合の前後の状態差分
- **ペルソナ破綻の傾向:** persona_deviation が低いDayに共通するパターン

---

## 分析手順

1. **ログファイルの読み込み:**
   `output/generation_log.json` を Read で読み込み、JSON構造を解析する。

2. **サマリの集計:**
   全体の成功率、リトライ数、フォールバック数を集計する。

3. **Day別の詳細分析:**
   各Dayの records を順に確認し、CriticScore、処理時間、リトライ回数を記録する。

4. **異常値の検出:**
   CriticScore が3未満（最終結果）、deviation が大きい、処理時間が平均の2倍以上のケースを検出する。

5. **傾向分析:**
   Day 1〜7 にかけてのスコア・処理時間の推移から傾向を読み取る。

6. **レポート生成:**
   分析結果を構造化したレポートにまとめる。

---

## 注意事項

- ログファイルが存在しない場合は、パイプライン未実行であることを報告する
- JSON パースエラーが発生した場合は、ログファイルの破損の可能性を報告する
- 分析はログデータに基づく客観的な事実のみを報告し、推測は「推測」と明記する
- 改善提案はプロンプト変更、パラメータ調整、コード修正の3カテゴリで分類する
