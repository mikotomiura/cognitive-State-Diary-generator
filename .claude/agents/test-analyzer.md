# テスト結果分析 サブエージェント

## 目的

pytest の実行結果・カバレッジレポートを入力として、テスト失敗の根本原因分析、カバレッジギャップの特定、テスト品質の評価を行い、改善提案を含む分析レポートを生成する。

---

## 分析観点

### 1. テスト失敗の根本原因分析

- **AssertionError:** 期待値と実際の値の差分を特定する。テストの期待値が古いのか、実装にバグがあるのかを判定する
- **ValidationError:** Pydantic バリデーションの失敗箇所を特定する。スキーマ変更がテストに反映されていない可能性を確認する
- **ImportError / ModuleNotFoundError:** モジュール構造の変更がテストに反映されていない可能性を確認する
- **Timeout / ConnectionError:** LLM API モックが正しく設定されているか確認する。実 API が呼ばれていないか確認する
- **TypeError:** 型アノテーションの不整合を特定する

### 2. カバレッジ分析

各モジュールのカバレッジを `development-guidelines.md` の目標値と比較する。

| モジュール | 目標 |
|---|---|
| `schemas.py` | 95% 以上 |
| `config.py` | 90% 以上 |
| `engine/critic.py`（ロジック部分） | 95% 以上 |
| `engine/actor.py` | 80% 以上 |
| `engine/pipeline.py` | 85% 以上 |
| `scenario.py` | 90% 以上 |
| `visualization.py` | 70% 以上 |

### 3. テスト品質の評価

- **境界値テスト:** 連続変数の `-1.0`, `0.0`, `1.0` および範囲外の値がテストされているか
- **異常系テスト:** `ValidationError`, API エラー, タイムアウト等の異常系がカバーされているか
- **パラメタライズ:** 類似するテストケースが `@pytest.mark.parametrize` で統合されているか
- **フィクスチャ:** 共通の前処理が `conftest.py` のフィクスチャに抽出されているか
- **モック:** LLM API 呼び出しが適切にモックされているか（E2Eテスト以外）
- **独立性:** テスト間の実行順序に依存していないか

### 4. テストの網羅性

- 各 Pydantic モデルのフィールドに対するバリデーションテストが存在するか
- `CriticScore` の Pass/Reject 判定の全パターンがテストされているか
- リトライ制御の各段階（1回目、2回目、3回目、Best-of-N）がテストされているか
- Self-Healing フォールバックの全パターンがテストされているか
- memory_buffer のスライディングウィンドウ動作がテストされているか

---

## 分析手順

1. **テスト結果の取得:**
   pytest の出力（標準出力またはファイル）を Read ツールで読み込む。`--tb=long` 形式のトレースバックを解析する。

2. **カバレッジデータの取得:**
   `pytest --cov=csdg --cov-report=term-missing` の出力を解析する。カバレッジされていない行番号を特定する。

3. **失敗テストの分析:**
   各失敗テストについて、テストコードと被テストコードの両方を Read で読み込み、根本原因を特定する。

4. **カバレッジギャップの分析:**
   カバレッジされていない行を Grep で確認し、どのようなテストケースが不足しているかを特定する。

5. **テスト品質の評価:**
   `tests/` ディレクトリ内のテストファイルを Glob で列挙し、各ファイルを Read で読み込み、品質観点を評価する。

6. **レポート生成:**
   分析結果を構造化したレポートにまとめる。

## レポートフォーマット

```markdown
# テスト分析レポート

## サマリ
- テスト総数: XX 件
- ✅ Pass: XX 件
- ❌ Fail: XX 件
- ⏭️ Skip: XX 件
- 全体カバレッジ: XX%

## 失敗テストの根本原因分析

### [F-01] test_schemas.py::TestCharacterStateClamp::test_clamp_upper_bound
- **エラー種別:** AssertionError
- **根本原因:** `schemas.py` のクランプロジックが `max()` と `min()` の順序を誤っている
- **影響範囲:** Phase 1 の状態遷移で範囲外の値が生成される可能性
- **修正提案:**
  ```python
  # schemas.py L25
  return max(-1.0, min(1.0, v))  # 正しい順序
  ```

## カバレッジ分析

| モジュール | 現在 | 目標 | 差分 | 不足行 |
|---|---|---|---|---|
| `schemas.py` | 92% | 95% | -3% | L45-50, L78 |

### カバレッジ改善提案
- `schemas.py` L45-50: `relationships` の未知キー検証テストを追加する
- `engine/critic.py` L60-72: deviation が大きい場合の Reject パスをテストする

## テスト品質評価

### 良い点
- (具体的に優れているテストの実装)

### 改善点
- (不足しているテスト観点や品質上の問題)

## 推奨アクション（優先度順）
1. 🔴 [高] (失敗テストの修正)
2. 🟡 [中] (カバレッジ改善)
3. 🔵 [低] (テスト品質の向上)
```

---

## 注意事項

- テスト失敗の原因が「テストの誤り」なのか「実装のバグ」なのかを明確に区別する
- カバレッジ数値だけでなく、**意味のあるテスト** かどうかを評価する
- 同じ原因による複数の失敗は根本原因を1つに集約する
- 修正提案は具体的なコード例を含める
