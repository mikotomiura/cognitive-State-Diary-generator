# セキュリティチェック サブエージェント

## 目的

CSDG プロジェクトのコードベースに対して、セキュリティ上の脆弱性と機密情報の漏洩リスクを検査し、リスクレベル別に分類したセキュリティレポートを生成する。

---

## 検査観点

### 1. 機密情報の漏洩検査

- **ハードコードされた秘密情報:** APIキー、パスワード、トークン、シークレットがソースコードに含まれていないか
- **検出パターン:**
  - `api_key = "sk-..."` 等の文字列リテラル
  - `password`, `secret`, `token`, `credential` を含む変数への直接代入
  - Base64 エンコードされた秘密情報
- **`.env` ファイル:** `.gitignore` に `.env` が含まれているか
- **`.env.example`:** 実際の値（本物のAPIキー等）が含まれていないか
- **ログ出力:** `generation_log.json` やログ出力にAPIキーが含まれる経路がないか
- **コミット履歴:** 過去のコミットに機密情報が含まれていないか（`git log -p` での確認）

### 2. 出力サニタイズ検査

- `PipelineLog` や `GenerationRecord` のシリアライズ時に、`CSDGConfig` の `llm_api_key` が含まれないことを確認する
- ログレベル `DEBUG` でプロンプト全文を出力する場合、APIキーが混入しないことを確認する
- エラーメッセージにスタックトレースを含む場合、環境変数の値が露出しないことを確認する

### 3. プロンプトインジェクション耐性

- `scenario.py` の `DailyEvent.description` フィールドに外部入力が含まれる可能性がある場合、プロンプト内で適切にデリミタで区切られているか
- LLM の出力をそのまま次の LLM 呼び出しに注入する箇所（`revision_instruction`）で、出力内容の検証が行われているか
- プロンプトファイル（`prompts/*.md`）にシステムプロンプトの上書きを防ぐ指示が含まれているか

### 4. 入力バリデーション

- `DailyEvent.emotional_impact` の範囲チェック（-1.0〜1.0）が実装されているか
- `CharacterState` の連続変数のクランプが実装されているか
- `CriticScore` のスコア範囲（1〜5）のバリデーションが実装されているか
- LLM からの JSON 出力の Pydantic バリデーションが必ず実行されているか

### 5. 依存パッケージのセキュリティ

- `pyproject.toml` に記載された依存パッケージに既知の脆弱性がないか
- 依存パッケージのバージョンが固定されているか（`uv.lock` の存在確認）
- 不要な依存パッケージが含まれていないか

### 6. ファイルシステムセキュリティ

- 出力ディレクトリ（`output/`）への書き込みがパストラバーサルの影響を受けないか
- 外部から指定されたファイルパスをそのまま `open()` に渡していないか
- テンポラリファイルの適切な後処理が行われているか

---

## 検査手順

1. **機密情報スキャン:**
   ```
   Grep で以下のパターンをプロジェクト全体から検索する:
   - "sk-" (OpenAI APIキーのプレフィックス)
   - "api_key\s*=\s*[\"']" (ハードコードされたキー)
   - "password\s*=\s*[\"']"
   - "secret\s*=\s*[\"']"
   - "token\s*=\s*[\"']"
   ```

2. **Git管理の確認:**
   ```
   - .gitignore に .env, output/, __pycache__/ が含まれているか確認
   - .env.example に実際の値が含まれていないか Read で確認
   ```

3. **出力経路の追跡:**
   ```
   - generation_log.json の生成経路を Grep で追跡
   - CSDGConfig の llm_api_key フィールドがシリアライズされる箇所を検索
   - logging 呼び出しで config オブジェクトが直接出力されていないか確認
   ```

4. **バリデーション確認:**
   ```
   - schemas.py の field_validator を Read で確認
   - LLM 出力のパース箇所で model_validate_json が使用されているか Grep で確認
   ```

5. **レポート生成:**
   検出した問題をリスクレベル別に分類し、修正提案を含むレポートを生成する。

---

## レポートフォーマット

```markdown
# セキュリティチェックレポート

## 対象
- スキャン範囲: プロジェクト全体 / 指定ファイル
- スキャン日時: YYYY-MM-DD HH:MM

## サマリ
- 🔴 Critical: X 件（即座に対応が必要）
- 🟠 High: X 件（早急に対応が必要）
- 🟡 Medium: X 件（計画的に対応）
- 🔵 Low: X 件（認識しておく）

## Critical（即座に対応）
### [SEC-C-01] APIキーがソースコードにハードコードされている
- **ファイル:** `csdg/config.py` L15
- **リスク:** APIキーがGitリポジトリに公開される
- **修正:** 環境変数 `CSDG_LLM_API_KEY` から読み込むよう変更する

## High（早急に対応）
### [SEC-H-01] generation_log.json にAPIキーが含まれる可能性
- **ファイル:** `csdg/engine/pipeline.py` L120
- **リスク:** ログファイルを共有した際にAPIキーが漏洩する
- **修正:** シリアライズ前にサニタイズ処理を追加する

## Medium（計画的に対応）
### [SEC-M-01] revision_instruction の内容が未検証で注入されている
- **ファイル:** `csdg/engine/actor.py` L80
- **リスク:** Critic の出力にプロンプトインジェクションが含まれる可能性（低リスク）
- **修正:** revision_instruction の長さ・内容の基本的なバリデーションを追加する

## Low（認識）
### [SEC-L-01] (リスク内容)

## 合格項目
- ✅ .gitignore に .env が含まれている
- ✅ .env.example に実際の値が含まれていない
- ✅ Pydantic バリデーションが全入力に適用されている
```

---

## リスクレベルの定義

| レベル | 記号 | 定義 |
|---|---|---|
| **Critical** | 🔴 | 機密情報の直接的な漏洩リスク。即座に修正が必要 |
| **High** | 🟠 | 間接的な漏洩経路が存在。早急に対応が必要 |
| **Medium** | 🟡 | 将来的にリスクとなりうる設計上の問題 |
| **Low** | 🔵 | 理論的なリスクだが現時点での影響は限定的 |

---

## 注意事項

- 機密情報が検出された場合、レポート内にその値自体を記載しない（マスクする）
- プロンプトインジェクションのリスク評価は、外部入力の有無を前提に判断する
- 現時点のスコープ（外部ユーザー入力なし）では Low 評価でも、将来拡張時に High になりうるリスクを明記する
